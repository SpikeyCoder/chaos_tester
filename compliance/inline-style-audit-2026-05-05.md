## 5. Refactor strategy

**Phase 1 — class library (`static/report.css`)**

- Pick the top 25 patterns from §3 and write them as utility classes. Each one deletes between 1 and ~30 inline-style occurrences.
- Group by intent: `.badge-{success,danger,warning,info,purple,pink}`, `.alert-{info,warning,danger}`, `.card`, `.card-flat`, `.status-card`, `.btn-reset`, `.sr-only`, `.u-display-{none,grid,flex}`, severity-pill family, alert family.
- Author the class library in one commit; it ships a stylesheet but does not touch any template, so it is safe to merge ahead of the template edits.

**Phase 2 — template-by-template substitution**

- Replace inline static `style="..."` attributes with the corresponding class. Refactor in priority order: `sample_report.html`, `report.html`, `dashboard.html`, then the long tail.
- Visual regression after each template ships: capture Playwright screenshots of `/sample-report` and a representative `/report/<run_id>` and diff against the baseline. The report renderer is a single Jinja path (`render_template('report.html', ...)`), so there is no second renderer to keep in sync.

**Phase 3 — dynamic-style refactor**

- For each row in §4, replace the interpolated style with a CSS custom property + class. Example:

```html
<!-- before -->
<div style="border-left:4px solid {{ pdata.color }};">...</div>

<!-- after -->
<div class="status-card" style="--status-color: {{ pdata.color }};">...</div>
```

```css
/* static/report.css */
.status-card {
  border-left: 4px solid var(--status-color);
  padding: 16px;
  border-radius: 8px;
}
```

Note: the **value** of `--status-color` is still interpolated, but it is now a single token (a hex / rgb color) instead of a full CSS declaration. A future CSP move to drop `style-src-attr 'unsafe-inline'` would still need an allowlist for these inline custom-property assignments — see Phase 5.

**Phase 4 — drop `'unsafe-inline'` from `style-src`**

- Update `app.py:131` once Phases 2 and 3 are done. Verify no CSP violations on every Flask route. Add a CI grep that fails any future PR that introduces a `style=` attribute under `templates/`.

**Phase 5 — (stretch) split into `style-src-elem` / `style-src-attr`**

- After Phase 4, optionally split the directive: `style-src-elem 'self'; style-src-attr 'self' 'unsafe-hashes' <hashes>`. This keeps the `--status-color` inline assignments working under explicit hashes rather than blanket `'unsafe-inline'`.

After Phase 4-B (this PR) merges and Cloud Run deploys, the live
`Content-Security-Policy` header on every response from
`https://website-auditor.io` no longer contains `'unsafe-inline'` on any

### Operational notes

- Inline-style hashes are refreshed on every Cloud Run deploy by the
  `Refresh CSP inline-style hashes` step (`compute_style_hashes.py --check`).
- Header byte budget is gated by `Check CSP header byte budget`
  (`check_csp_header_size.py`, default `--max 8100`). Any future PR that
  adds an inline style and pushes the header over budget fails CI before
  deploy — preventing a recurrence of the silent-strip incident that
  delayed Phase 4-A.
- The `static/css/utilities.css` library is the canonical home for new
  styling. New templates should use existing utility classes; novel
  patterns should be added to `utilities.css` (and to the substitution
  table in `outputs/refactor_sample_report.py` if developer batch
  refactoring is wanted).

### Browser support

CSP3's `style-src-elem` and `style-src-attr` directives require:
- Chrome 75+ (Jul 2019)
- Firefox 77+ (Jun 2020)
- Safari 15.4+ (Mar 2022)

Older browsers (~3% of traffic) fall back to `default-src 'self'` for
all style directives, which has the same effect as the previous
`style-src 'self' 'unsafe-inline'` *minus* the `'unsafe-inline'` —
i.e. these clients see broken inline styles. This is the deliberate
trade-off that closes the Medium finding.
