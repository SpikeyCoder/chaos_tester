Authentication

All API requests require an API key sent via the Authorization header as a Bearer token.

Getting Your API Key

  1. Sign up or log in to api.website-auditor.io
  2. Navigate to the API Keys section in your dashboard
  3. Create a new API key (you can have multiple keys)
  4. Use this key in all API requests

Example Request

curl -X POST https://api.website-auditor.io/run \ -H "Authorization: Bearer YOUR_API_KEY" \ -H "Content-Type: application/json" \ -H "X-Requested-With: XMLHttpRequest" \ -d '{"base_url":"https://example.com"}'

Endpoints

POST
/run
Launch a new audit
⌄

Request

Send a JSON payload with audit parameters.

Headers:

Authorization: Bearer <API_KEY> Content-Type: application/json X-Requested-With: XMLHttpRequest

Body Parameters:

Parameter Type Description
base_url string Website URL to audit required
environment string One of: staging, test, production (default: production)
max_pages integer Max pages to crawl (1-1000, default: 100)
crawl_depth integer Link depth to follow (1-10, default: 3)
request_timeout integer Request timeout in seconds (1-120, default: 30)
concurrency integer Parallel requests (1-20, default: 5)
run_availability boolean Check HTTP status & availability (default: true)
run_links boolean Check for broken links (default: true)
run_forms boolean Test form submission (default: true)
run_chaos boolean Run chaos engineering tests (default: false)
run_auth boolean Test authentication flows (default: false)
run_security boolean Run security scans (default: true)

Response

Status: 202 Accepted

The audit is queued and will begin processing.

{ "status": "started", "run_id": "run_abc123xyz", "started_at": "2026-04-15T14:30:00Z", "base_url": "https://example.com" }

Example Request:

{ "base_url": "https://example.com", "max_pages": 50, "crawl_depth": 2, "run_security": true, "run_chaos": false }
GET
/api/status
Check current run status
⌄

Request

Get the status of the current or most recent audit run.

Headers:

Authorization: Bearer <API_KEY>

Query Parameters: None

Response

Status: 200 OK

{ "status": "running", "current_run_id": "run_abc123xyz", "progress": { "pages_crawled": 12, "total_pages": 50, "pct_complete": 24, "current_module": "links" }, "started_at": "2026-04-15T14:30:00Z" }

Possible Status Values:

  • idle: No audit running
  • running: Audit in progress
  • completed: Audit finished successfully
  • failed: Audit encountered an error

    GET
    /stream
    Real-time progress via Server-Sent Events (SSE)
    ⌄

    Request

    Open a Server-Sent Events (SSE) stream to receive real-time progress updates for the current audit.

    Headers:

    Authorization: Bearer <API_KEY> Accept: text/event-stream

    Response

    Status: 200 OK

    Content-Type: text/event-stream

    Streams events with this structure:

    data: { "module": "links", "pct": 45, "msg": "Checking links...", "ts": "2026-04-15T14:32:10Z" }

    JavaScript Example:

    const es = new EventSource( '/stream', { headers: { Authorization: 'Bearer YOUR_KEY' } } ); es.onmessage = (e) => { const progress = JSON.parse(e.data); console.log(progress.msg, progress.pct + '%'); };

    HTTP Status Codes

    The API uses standard HTTP status codes to indicate the result of your request.

    200 OK
    Request succeeded and data is returned
    201 Created
    Resource was successfully created
    202 Accepted
    Request accepted for processing (audit queued)
    400 Bad Request
    Invalid request parameters
    401 Unauthorized
    Missing or invalid API key
    403 Forbidden
    Quota exceeded or rate limited
    404 Not Found
    Resource not found (e.g., invalid run_id)
    429 Too Many Requests
    Rate limit exceeded (see Rate Limits section)
    500 Internal Server Error
    Server-side error; retry with exponential backoff
    503 Service Unavailable
    Service temporarily unavailable; retry after delay

    Best Practices

    Error Handling

    All error responses include a JSON body with details:

    { "error": "invalid_request", "message": "base_url is required", "code": 400 }

    Polling vs Streaming

    For progress updates, prefer Server-Sent Events (SSE) via /stream rather than repeatedly polling /api/status. SSE is more efficient and provides real-time updates without extra API calls.

    Retry Strategy

    Use exponential backoff for retries on 5xx errors. Start with 1 second and double the delay with each retry (1s, 2s, 4s, 8s, etc). Maximum 5 retries recommended.

    Request Timeouts

    Set your HTTP client timeout to at least 60 seconds, as audit processing may take time depending on site size and concurrency settings.

    Webhooks (Coming Soon)

    We're building webhook support to notify your system when audits complete. This will eliminate the need for polling entirely. Stay tuned!
